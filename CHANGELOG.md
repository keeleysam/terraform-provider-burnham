## [Unreleased]

### Added
- New family **regex**: PCRE-flavored regular expressions with the features Terraform core's RE2 engine omits (backreferences, lookahead/lookbehind). `pcre_match` (bool), `pcre_captures` (map of numbered + named groups of the first match), `pcre_find_all` (all matches), `pcre_replace` (with `$1` / `${name}` in the replacement), and `pcre_split`. Backed by [fancy-regex](https://github.com/fancy-regex/fancy-regex) compiled to `wasm32-wasip1` and run under the pure-Go [wazero](https://github.com/tetratelabs/wazero) runtime, so the provider stays CGO-free and results are deterministic, with bounded backtracking so a pathological pattern errors rather than hanging the plan. The wasm engine is gitignored and produced by `go generate`.
- New family **image**: `svg_render` rasterizes an SVG document to a base64 PNG at near-browser fidelity (gradients, `gradientTransform`, `clipPath`, masks, filters, patterns, text, and native color emoji). Rendering is done by [resvg](https://github.com/linebender/resvg) (Apache-2.0 OR MIT) compiled to `wasm32-wasip1` and executed under the pure-Go [wazero](https://github.com/tetratelabs/wazero) runtime, so the provider stays CGO-free and output is byte-identical across host architectures (WASM float semantics are deterministic). Fonts come only from the bundled set plus any caller-supplied base64 TTF/OTF via the `fonts` option; no system fonts are consulted, keeping renders deterministic and self-contained. The bundle covers the common cases: the generic families `sans-serif`/`serif`/`monospace` map to the Noto Sans/Serif/Sans Mono variable builds (so any `font-weight` and italic render from one file per family), the web-safe named fonts (Arial, Helvetica, Times New Roman, Courier New, Verdana, Georgia, ...) are recognized and rendered with the metric-compatible Liberation families via an alias map, and color emoji use the Noto Color Emoji COLRv1 vector build. Options: `width` / `height` (supply one for aspect-preserving scale, both for an exact box, neither for the intrinsic size), `scale`, and `fonts`. The wasm blob and emoji font are gitignored and produced by `go generate` (a Rust build and a pinned font download), so neither is committed to the repo.
- New family **color**: pure, deterministic color functions for the places infra config carries colors (GitHub/GitLab label backgrounds, Grafana/Datadog series, generated themes). `color_convert` parses any CSS Color 4 notation (hex, `rgb()`, `hsl()`, `hwb()`, `lab()`/`lch()`, `oklab()`/`oklch()`, named) and re-serializes it as `hex`, `rgb`, `hsl`, or `oklch`, with `{ hash, uppercase }` options for the `#`-vs-no-`#` mismatch between providers. `color_contrast_ratio` returns the [WCAG 2.x](https://www.w3.org/WAI/WCAG22/Understanding/contrast-minimum.html) ratio (1-21) from the exact relative-luminance formula, and `color_readable_text` picks the most legible text color (black/white by default, or a custom candidate list) for a background. `color_distinct` generates N maximally-distinct colors by sweeping the OKLCh hue wheel at fixed lightness/chroma, deterministically so plans never churn. `color_mix` blends two colors (OKLCh default, mirroring CSS `color-mix`), `color_ramp` interpolates N colors across a list of stops, and `color_adjust` nudges OKLCh channels with a chroma.js-style grammar (`{ lightness = "*0.9" }` to darken, `{ hue = "+30" }` to rotate, `{ chroma = "0" }` to grayscale), collapsing lighten/darken/saturate/desaturate/rotate-hue/fade into one function. `color_scheme` rotates a base color's OKLCh hue into a harmony palette (`complementary`, `analogous`, `triadic`, `split-complementary`, `tetradic`, `square`, with a tunable `angle` for the analogous and split schemes), and `color_nearest` snaps an arbitrary color onto a fixed palette by perceptual distance (CIEDE2000 by default, or `oklab`/`lab`/`rgb`), returning the matched entry exactly as written. All math runs in the perceptually-uniform OKLab/OKLCh space via [go-colorful](https://github.com/lucasb-eyer/go-colorful) with CSS parsing by [csscolorparser](https://github.com/mazznoer/csscolorparser); both pure-Go, no CGO.
- Cryptography expansion (JOSE): a deterministic JWT/JWS and JWK stack. `jwt_sign` mints a compact JWS/JWT and is byte-stable across every algorithm it offers: HS256/384/512 (HMAC), ES256 (ECDSA P-256 signed via the existing RFC 6979 deterministic `k`, emitted as the fixed 64-byte `R||S` per [RFC 7518](https://www.rfc-editor.org/rfc/rfc7518) rather than DER), EdDSA ([RFC 8037](https://www.rfc-editor.org/rfc/rfc8037), Ed25519), and RS256/384/512 (RSASSA-PKCS1-v1_5). Time claims are never derived from the clock: `exp` / `iat` / `nbf` are whatever the caller supplies. `jwt_decode` returns `{ header, payload }` without verifying the signature. `jwt_verify` returns `{ valid, header, payload }`, always checks the signature, and only validates `exp` / `nbf` when `options.now` (unix seconds or an RFC 3339 string) is supplied, so it stays pure by default; `options.algorithm` pins the accepted alg to guard against algorithm substitution. `jwk_encode` / `jwk_decode` convert a PEM key (EC, RSA, Ed25519) to and from a JWK object (round-trip pair), `jwk_thumbprint` computes the [RFC 7638](https://www.rfc-editor.org/rfc/rfc7638) canonical thumbprint (the standard `kid`) from a PEM string or a JWK object, and `jwks` assembles a JWK Set from a list of PEM strings or JWK objects. JWK handling is backed by [go-jose/go-jose/v4](https://github.com/go-jose/go-jose); the compact JWS is hand-rolled so determinism (in particular the ES256 `R||S` encoding reusing `detECDSASigner`) is fully controlled. RSASSA-PSS is intentionally omitted because its signatures are randomised. Verified against the RFC 7515 A.1 (HS256), RFC 8037 A.4 (Ed25519), and RFC 7638 §3.1 (thumbprint) known-answer vectors.
- Structured-data expansion: `json_canonicalize`: [RFC 8785](https://www.rfc-editor.org/rfc/rfc8785) JSON Canonicalization Scheme (JCS), the exact bytes to feed `hmac` / `hkdf` / `pkcs7_sign` so the same logical value always hashes identically. Sorts object keys, strips insignificant whitespace, and serializes numbers under the I-JSON / ES6 double rules (so integers beyond 2^53 follow the spec's double serialization, documented on the function). Backed by [gowebpki/jcs](https://github.com/gowebpki/jcs).
- Expression Languages expansion (PromQL): `promqlencode` builds a Prometheus query from an HCL data tree mirroring the PromQL AST (selectors, matchers, ranges, offset/@, calls, aggregations, binary ops with vector matching, subqueries, unary `neg`/`pos`, plus a `{ raw = "..." }` escape), with label values quoted correctly so there is no fragile interpolation. `promqldecode` parses a query back into that data tree, so `promqlencode(promqldecode(q))` round-trips to the canonical form of `q` (the implicit `__name__` matcher folds into the `name` field, and every node is fully structured with no `raw` fragment). `promqlvalidate` reports whether a query is valid (a bool; the parser type-checks, so type errors are caught too). `promqlformat` returns the canonical query, or the multi-line indented form with `{ pretty = true }`. Output is parsed back before returning, so `promqlencode` never emits an invalid query and it is byte-identical to `promqlformat`. Backed by [prometheus/prometheus](https://github.com/prometheus/prometheus)'s own parser, so a query that validates here is valid in Prometheus.
- New family **Expression Languages** (Cedar): `cedarencode` / `cedardecode` convert a Cedar authorization policy between its human-readable DSL and canonical EST JSON forms (so `cedarencode(cedardecode(x))` round-trips), `cedarvalidate` / `cedarformat` check and canonicalize a policy document, and `cedarevaluate` authorizes a request (principal, action, resource, context, entities) against it and returns the decision. Backed by [cedar-go](https://github.com/cedar-policy/cedar-go), the official implementation, so a `cedarevaluate` decision is authoritative: it matches the engine Amazon Verified Permissions runs. Grouped with the existing `cel*` and `oel*` functions under the Expression Languages family.
- New family **compression**: `base64zopfli` (RFC 1952 gzip via [Zopfli](https://github.com/google/zopfli)'s iterative DEFLATE encoder, a drop-in, tighter replacement for the built-in `base64gzip`; gzip header pinned to `MTIME=0` / `XFL=2` / `OS=255` for deterministic, portable output, with an optional `{ iterations }` knob, default 15) and `base64brotli` (RFC 7932 Brotli, ~8–10% smaller than `base64gzip` on text, with optional `{ quality, lgwin }`, defaults 11 / 22). Both pure-Go (`CGO_ENABLED=0`, via `foobaz/go-zopfli` and `andybalholm/brotli`) and deterministic for plan stability. The RFC 7932 §10 `mode` hint is intentionally not exposed: the pure-Go encoder reads it only to special-case `font`, so `text`/`generic` are byte-identical and a `mode` option would be a no-op.
- New family **cryptography**: `hmac` (RFC 2104), `hkdf` (RFC 5869), `pem_decode` (RFC 7468), `x509_inspect`, `x509_fingerprint`, `csr_inspect` (PKCS#10), `asn1_decode` (BER/DER walk), plus a deterministic signing pipeline supporting both ECDSA P-256 and Ed25519: `ecdsa_p256_key_from_seed` (HKDF-SHA256 seed → secp256r1 scalar → PKCS#8 PEM), `ed25519_key_from_seed` (HKDF-SHA256 seed → 32-byte Ed25519 seed → PKCS#8 PEM per RFC 8032 §5.1.5), `x509_self_sign` (RFC 5280 v3 self-signed cert: ECDSA via RFC 6979 deterministic `k`, Ed25519 via PureEdDSA naturally deterministic; serial bounded to 20 octets per §4.1.2.2, UTCTime/GeneralizedTime split at 2050 per §4.1.2.5, BasicConstraints critical with cA=FALSE per §4.2.1.9, Ed25519 algorithm encoding per RFC 8410), and `pkcs7_sign` (RFC 5652 SignedData ContentInfo, encapsulated `id-data`, no signed attributes per §5.3, embedded cert, byte-stable across runs: ECDSA signs SHA-256(data), Ed25519 signs raw data with `id-sha512` digest OID per RFC 8419 §3). The chain composes but `pkcs7_sign` also accepts caller-supplied real-world identities for the CA-issued case and rejects mismatched key/cert pairs at call time rather than producing silently-unverifiable output. macOS configuration-profile signing requires ECDSA P-256; Ed25519 is for non-Apple consumers (OpenSSL `cms`, container signing).
- New family **identifiers**: deterministic `uuid_v5` (RFC 9562 §5.5), deterministic `uuid_v7` (RFC 9562 §5.7), `uuid_inspect`, deterministic `nanoid`, deterministic `petname`.
- New family **text**: `unicode_normalize` (UAX #15 NFC/NFD/NFKC/NFKD), `slugify`, `levenshtein`, `wrap`, `cowsay`, `qr_ascii`.
- New family **geographic**: `geohash_encode`, `geohash_decode`, `pluscode_encode`, `pluscode_decode` (Open Location Code).
- Numerics expansion: `mean`, `median`, `mode`, `percentile`, `variance`, `stddev`, `clamp`, `mod_floor`, `gcd`, `lcm`. `gcd` and `lcm` operate on a `list(number)` of integers using arbitrary-precision `math/big`: `gcd` folds `(*big.Int).GCD` (non-negative result, any sign), `lcm` combines pairwise as `l / gcd(l, x) * abs(x)` so the result stays exact for very large inputs. Conventions: `gcd` of all zeros is 0, `gcd(0, n)` is `abs(n)`, `lcm` is 0 when any element is 0, and a single-element list returns the absolute value. Both reject an empty list and any non-integer element (naming the offending value).
- Numerics expansion (bitwise operations): `bit_and`, `bit_or`, `bit_xor` (each folded over a non-empty `list(number)`), `bit_not` (width-parameterized complement: `value ^ (2^bits - 1)`, requiring `bits >= 1` and `0 <= value < 2^bits`, since a width-less NOT is infinite in two's-complement), `bit_shift_left` / `bit_shift_right` (the latter an arithmetic shift that floors a negative value toward negative infinity), `popcount` (Hamming weight; a negative value errors because it has infinitely many set bits in two's-complement), and single-bit helpers `bit_set` / `bit_clear` / `bit_test`. Terraform's configuration language has no bitwise operators or functions at all, so these fill that gap. Every function is integer-only and rejects a non-integral or infinite argument, and all arithmetic is arbitrary-precision `math/big`, so nothing overflows `int64` (`bit_shift_left(1, 100)` and `popcount(2^64)` are exact). AND/OR/XOR treat a negative operand as an infinite two's-complement bit string, matching `math/big`; the flag / mask use case uses non-negative integers.
- Network expansion: `pigeon_throughput`: RFC 1149 / RFC 2549 IP-over-Avian-Carriers throughput calculator.
- Network expansion: `ip_idunno_encode` / `ip_idunno_decode`: RFC 8771 Internationalized Deliberately Unreadable Network Notation. Dual-stack, reaches the §4.1 Minimum Confusion Level on every output, and reproduces the §5 worked example bit-for-bit.
- New family **encoding**: hex and base64 byte codecs ([RFC 4648](https://www.rfc-editor.org/rfc/rfc4648)). `hexencode` / `hexdecode` add the hex decode Terraform core lacks, closing the gap that left `hmac` / `hkdf` telling callers to base64-wrap hex keys; `hexdecode` is case-insensitive and ignores ASCII whitespace. `base64encode` takes an options object for the RFC 4648 variants: `{ url_safe }` for the §5 URL- and filename-safe alphabet (JWT/JOSE, OAuth PKCE), `{ padding = false }` for the raw unpadded form. With no options it is identical to core `base64encode`. `base64decode` is deliberately lenient: it accepts either alphabet, padded or not, ignoring ASCII whitespace, so it is a friction-free superset of core's stricter decoder.
- Query and Patch expansion: `jq`: evaluate a [jq](https://jqlang.github.io/jq/) program against a decoded value via pure-Go [itchyny/gojq](https://github.com/itchyny/gojq), the expressive sibling of `jmespath_query` / `jsonpath_query`. A jq program is a stream, so the result is always a list (one element per value produced); an optional `{ vars }` object binds jq variables (`$name`). The full jq language is available, including the time builtins: `now` / `localtime` work but are non-deterministic, so programs deriving from them will churn the plan (documented on the function). `env` / `$ENV` return an empty object (the host process environment is not exposed) and `input` / `inputs` error (there is no secondary input stream).
- Query and Patch expansion: `jsonata_query` / `jsonata_validate`: evaluate a [JSONata](https://jsonata.org/) expression against a decoded value via pure-Go [recolabs/gnata](https://github.com/recolabs/gnata) (JSONata 2.x, the language behind AWS Step Functions' JSONata mode), the transformation-oriented sibling of `jmespath_query` / `jsonpath_query`, with query, aggregation, mapping, object/array construction, and lambdas. Because burnham is a pure provider, the three non-deterministic JSONata builtins (`$now`, `$millis`, `$random`) are overridden to fail, so an expression using them errors rather than churning the plan; numbers follow JSONata's IEEE-754 double model, so a value passed through keeps full precision while a value computed on can round beyond 2^53 (documented on the function). `jsonata_validate` reports whether an expression is syntactically valid (a bool), never fails the plan (oversized input returns `false`), and treats an expression that references the non-deterministic builtins as valid since validity is about syntax.
- Cryptography expansion: `btoe` / `etob`: [RFC 1751](https://www.rfc-editor.org/rfc/rfc1751) "A Convention for Human-Readable 128-bit Keys" (S/Key). The functions carry the RFC's own names: `btoe` (bytes-to-english) encodes a key (hex, length a multiple of 8 bytes) as six dictionary words per 64-bit block, and `etob` (english-to-bytes) reverses it, verifying the two parity bits the RFC appends so a transcription error is caught. A faithful port of the RFC appendix's `extract` / `insert` / parity / `standard()` routines and 2048-word dictionary, verified against the three worked examples in the RFC body.
- Text expansion: `parse_kv`: parses a delimited key/value string into a `map(string)`, a robust replacement for the fragile HCL idiom `{ for p in split(",", s) : split("=", p)[0] => split("=", p)[1] }`. Each pair is split on the first `kv_sep` only (so a value may contain the separator, `a=b=c` gives `{ a = "b=c" }`), surrounding whitespace is trimmed, and parsing is quote-aware by default so a `pair_sep`/`kv_sep` inside `"..."` or `'...'` is literal (`a="x,y",b=2` gives `{ a = "x,y", b = "2" }`). Options: `pair_sep` (default `,`), `kv_sep` (default `=`), `trim` (default true), `unquote` (default true). Empty segments are skipped, a pair with no `kv_sep` is an error, and a duplicate key is an error (a map cannot hold duplicates). Values are always strings, matching the naive workaround's output.
- Text expansion: `dedent`: removes the longest run of leading whitespace common to all non-blank lines (the `textwrap.dedent` operation), the inverse of Terraform's built-in `indent`. Relative indentation is preserved; whitespace-only lines don't constrain the common margin and are normalized to empty. Useful for left-aligning indented scripts/manifests pulled from a variable or a plain (non-indented) heredoc.
- Encoding expansion: `base32encode` / `base32decode` ([RFC 4648](https://www.rfc-editor.org/rfc/rfc4648)): Terraform core has no base32 codec at all (a core PR was closed as "belongs in a provider-defined function"). `base32encode` takes `{ hex_alphabet }` (standard `A–Z2–7` vs extended-hex `0–9A–V` for NSEC3) and `{ padding }`; with no options it is standard + padded. `base32decode` is lenient (case-insensitive, padding optional, ASCII whitespace ignored) and takes `{ hex_alphabet }`: the alphabet can't be auto-detected because the two overlap. The headline use is TOTP/MFA secrets (unpadded standard base32). Completes burnham's RFC 4648 family (16/32/64).
- Encoding expansion: `urlencode` (with options) / `urldecode`: `urlencode` takes `{ mode }`: `query` (default, `application/x-www-form-urlencoded`, byte-identical to core `urlencode`), `path` (RFC 3986 path segment, space → `%20`), or `component` (strict: only unreserved characters survive). `urldecode` is **the percent-decoder Terraform core lacks entirely**; it takes the same `{ mode }` because `+` is ambiguous, a space in a query string but a literal `+` in a path, so `query` decode maps `+` → space while `path`/`component` leave it literal (both always decode `%XX`).
- `cmd/gendoctemplates` writes per-function `subcategory:` headers so the registry sidebar groups functions by family.

### Changed
- `jsonencode` no longer HTML-escapes `<`, `>` and `&` by default, so they are written as literal characters, which is what a pretty-printer for human-reviewed output and `local_file` writes almost always wants (Terraform's built-in `jsonencode` and Go's encoder escape them to `<` / `>` / `&`). Pass the new `{ escape_html = true }` option to restore the escaped form, e.g. when embedding JSON in an HTML `<script>` context. Output for values containing `<`/`>`/`&` therefore changes.
- `hujsonencode` matches `jsonencode`'s new default: `<`, `>` and `&` are written literally rather than HTML-escaped. Its expanded (default) path previously escaped them via `json.Marshal`; the compact path already emitted them literally because `hujson.Format` normalizes `\uXXXX` escapes back to characters. Pass `{ escape_html = true }` to escape them to their `\uXXXX` forms on both paths. On the compact path this re-escapes the packed output, restricting the rewrite to string literals so `//` and `/* */` comments are left untouched.
- Provider Schema description rewritten to enumerate all nine families.
- README adds a "Tagged-value helpers" sub-table covering `plistdata`/`plistdate`/`plistreal` and `regbinary`/`regdword`/`regexpandsz`/`regmulti`/`regqword`, which were registered but not previously documented in the README.
- All `MarkdownDescription` strings across `dataformat/`, `network/`, and `transform/` collapsed to single literals (no mid-paragraph `+` joins).
- `geohash_decode` parameter renamed `hash` → `code` to match `pluscode_decode` and the broader geographic-family naming.
- `asn1_decode`'s `children` field is now `dynamic` (a tuple at runtime) instead of `list(dynamic)`. Tuples in HCL still accept `children[i]`, `length(children)`, and `[for x in children : ...]`, but `tolist(children)` and any type-assertion that explicitly demanded `list(...)` will fail. This was forced by the heterogeneous-children panic fix (see Fixed); homogeneous-children HCL that relied on list typing should `[for x in children : x]` to coerce.
- HMAC and HKDF docstrings now share a single `hclByteHandlingGotcha` helper so the byte-handling explanation cannot drift between the two functions.
- CI now runs the full acceptance suite with `TF_ACC=1` and `-race`.

### Fixed
- `asn1_decode` no longer panics when decoding ASN.1 structures with heterogeneous children (e.g. CMS SignedData's SET children: SEQUENCE, OCTET STRING, [0]-tagged blobs). The decoder's `children` field is now a `Dynamic`-typed tuple instead of `list(dynamic)`, sidestepping cty's "inconsistent value types in ListVal" panic; HCL accessor syntax (`children[0]`, `length(children)`) is unaffected.
- `nanoid` no longer panics on a 256-codepoint alphabet (`byte(256)` overflow → `% 0` divide-by-zero); modulus arithmetic is now in `int`.
- `regdword` / `regqword` now reject negative, fractional, and out-of-range inputs explicitly. Previously `(*big.Float).Uint64()` silently saturated negatives to `0` and overflow to `MaxUint*`.
- `geohash_encode` rejects exact `lat == 90` / `lon == 180` (upstream encoder wrapped these to the opposite corner) and `geohash_decode` shrinks the corner-cell bbox edges below the wrap threshold so feeding `lat_max` / `lon_max` back into the encoder round-trips into the same cell.
- ASN.1 decoder now bounds total node count (≤ 100,000), input length (≤ 8 MiB), and recursion depth (≤ 64 levels) to defeat adversarial blobs that would otherwise OOM the provider.
- CBOR decoder sets `MaxNestedLevels`, `MaxArrayElements`, `MaxMapPairs` and CBOR/MessagePack/VDF/KDL all bound input length to 16 MiB; `convert.go`'s `goToTerraformValue` / `terraformValueToGo` now cap recursion at 1024 levels.
- Cowsay rejects non-printable runes in `eyes` / `tongue` (no more ANSI escape smuggling) and caps the input message at 64 KiB.
- Pluscode docstring example output corrected to the canonical `849VQHFJ+X6` for `(37.7749, -122.4194, 10)`.
- Provider Schema no longer mis-credits Pi to Chudnovsky; the plan-time path uses an embedded DPD-packed table.
